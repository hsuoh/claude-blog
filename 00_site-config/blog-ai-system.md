## 前提：このサイトの設計思想

- **サイト名**: 薬剤師キャリアナビ（pharmirai.com）
- **コンセプト**: 転職を煽らない。データから判断させる。
- **最終ゴール**: 収益構造を理解し、合理的に転職メリットがある薬剤師を転職サイトにつなぐ
- **ターゲット**: 20〜30代の病院・調剤薬局・ドラッグストア薬剤師
- **運営体制**: PM（あなた）＋ パートナー（WP入稿・装飾）＋ Claude（AI担当）
- **参考設計**: リベ大ブログ教室（収益化設計）× トピッククラスター戦略（SEO構造）

## ① AIチーム構成

| 役割 | 担当 | 主な責務 |
|---|---|---|
| **PM** | あなた（人間） | 方針決定・記事承認・公開判断・最終品質管理 |
| **リサーチAI** | Claude | KW調査・競合分析・ニーズ収集・データ収集 |
| **ライターAI** | Claude | 構成作成・本文執筆・見出し設計・ディスクリプション生成 |
| **SEO設計AI** | Claude | 内部リンク設計・カテゴリ分類・収益導線設計 |
| **データ分析AI** | Claude | 決算データ整理・調剤報酬改定解説・業界ニュースまとめ |
| **WP入稿担当** | パートナー（人間） | WP装飾・アイキャッチ作成・公開設定 |

**運用の基本原則**:
1. 記事1本につき「KW選定 → リサーチ → 構成 → 執筆 → 推敲・SEOチェック」をClaudeが一気通貫で行う
2. PMが方向性・KW・タイトルを承認してから執筆に入る
3. 執筆・推敲後はPMが一次レビュー → CoworkでWP下書き保存 → パートナーが装飾・公開設定 → PMが最終確認して公開

## ② 役割定義

### PM（プロジェクトマネージャー）の役割
- 月次の記事テーマ優先順位を決定します
- AIが出した構成案・タイトル案を承認・修正します
- 現役薬剤師として専門的な事実確認を行います
- 転職サイトアフィリエイトの掲載タイミングを判断します

### Claude（リサーチ・ライターAI）の役割
- KW候補をWeb検索で整理します（ラッコキーワードのデータも受け付けます）
- 調剤報酬改定の厚労省資料・業界ニュースを要約します
- 大手薬局の決算情報を整理して記事素材を作ります
- 競合上位記事の見出し構成を分析してギャップを見つけます
- 読者が「転職すべきか・残るべきか・様子見か」を判断できる記事を書きます
- 転職を煽らず、データと収益構造で読者が自己判断できる文章にします
- 各記事に必ずディスクリプション・タイトルタグ候補を3案出します
- 執筆後は必ずSEO・文脈・読みやすさの自己チェックを行い、推敲してから提出します

### パートナー（WP入稿担当）の役割
- CoworkがWPに下書き保存した記事を受け取ります
- 装飾メモを見ながらSWELLで吹き出し・ボックス・CTAを設置します
- アイキャッチ画像を作成・設定します
- スマホプレビューで崩れがないか確認します
- PMに確認依頼を送ります

## ③ 記事候補選定手順

### 記事の3分類（リベ大方式）

| 分類 | 役割 | 対応フォルダ |
|---|---|---|
| **まとめ記事（ピラー）** | テーマ全体を網羅・詳細記事への入口 | `01_pillar` |
| **集客記事（クラスター）** | ロングテールKWで流入を集める・まとめ記事に誘導 | `02_articles` / `05_stock` |
| **キラーページ（収益）** | 転職サイト比較・ランキング・直接的なCTA | `02_articles`（収益特化） |

データ記事（`03_data`）はまとめ記事・集客記事の両方を支える素材として機能します。

### 選定手順

```
STEP 1: ターゲットの悩みを3軸で整理する
└ ① お金の悩み（給与・ボーナス・昇給が止まった）
└ ② 職場の悩み（人間関係・残業・待遇が悪い）
└ ③ 将来の不安（調剤報酬改定・薬剤師余剰・AI）

STEP 2: 各悩みをキーワードに変換する
└ 「薬剤師 給料 下がる」「調剤薬局 将来性」など

STEP 3: 記事分類とフォルダを決める
└ まとめ記事（01）/ 集客記事（02・05）/ データ記事（03）/ キラー（02）

STEP 4: 不要な記事を明確に除外する
└ 収益ゴール（転職サービス登録）と無関係な記事は書かない
└ 例：「薬剤師 趣味」「薬剤師 勉強法」などは対象外

STEP 5: PMが優先順位を決定して月次計画に入れる
```

**選定の判断基準（優先度順）**:
1. 転職判断・給与・収益構造に直接関連するテーマ
2. 検索ボリューム 100〜3,000（競合が弱い帯を狙う）
3. 現役薬剤師として専門性で差別化できるテーマ
4. 調剤報酬改定・決算データで裏付けできるテーマ

## ④ キーワード選定手順

**使用ツール**:
- Claude（Web検索）：サジェスト・競合分析・ニーズ収集（ラッコキーワードの代替）
- Googleキーワードプランナー：検索ボリューム確認（PMが確認）
- Google検索：競合・検索意図の確認（無料）

```
STEP 1: シードKWを決める
└ site-profile.mdの「主要SEOキーワード」リストを起点にする

STEP 2: ClaudeがWeb検索でサジェストを展開する
└ 検索意図ごとにグループ分け
└ 競合上位記事の構成も分析する

STEP 3: PMがGoogleキーワードプランナーでボリュームを確認する
└ 狙い目：月間100〜3,000（新規サイトは100〜1,000から）

STEP 4: Googleで実際に検索して競合を確認する
└ 個人ブログ・中小メディアが多い → 勝てる（◎）
└ 大手転職サイト・求人サイトが多い → 難しい（△）
└ 厚労省・公式機関・大手企業ばかり → 避ける（✕）

STEP 5: 検索意図を分類してフォルダを決める
└ 情報収集型（know）→ 05_stock / 03_data
└ 比較・判断型（do） → 02_articles / 01_pillar
└ 転職意欲型（go） → 02_articles（収益記事）

STEP 6: タイトルKWを確定してPMが承認する
└ Claudeにタイトル案3つを出させてPMが選ぶ
```

**KWの選定ルール**:
- 1記事 = メインKW1つだけ（カニバリ防止）
- 共起語・関連語は本文中に自然に含める
- 新規サイトはまずロングテール（3語以上・月間100〜1,000）から始める

## ⑥ 構成作成手順

### リード文の必須要素
リード文には以下の4要素を必ず入れます：
1. **想定読者の悩み**（「〜でお悩みではないですか？」）
2. **悩みの結論・解決策**（冒頭で先出し）
3. **解決できる理由・権威性**（現役薬剤師・収益構造データをもとに）
4. **読後のベネフィット**（「この記事を読めば〜が判断できます」）

### Claudeへの構成作成依頼テンプレート

```
【構成作成依頼】
メインKW: [KW]
フォルダ: [01_pillar / 02_articles / 03_data / 05_stock]
ターゲット読者: [病院 / 調剤薬局 / ドラッグストア] 薬剤師
読者の悩み: [具体的な悩みを記入]
記事の結論: [転職すべき / 残るべき / 様子見 / 知識習得]
参考データ: [調剤報酬改定の○○ / ○○社の決算 / など]

出力:
- タイトル案 3つ（32文字以内・メインKW含む）
- ディスクリプション 2案（120文字以内）
- リード文の構成要素（4要素に沿って）
- H2見出し（5〜8個）
- 各H2のH3（2〜4個）
- 想定文字数
- 収益CTA設置箇所
- 内部リンク候補（設置箇所と紐づく記事名）
```

## ⑦ 記事フォーマット（統一仕様）

### 文体・スタイル

| 項目 | 方針 |
|---|---|
| 文体 | **ですます調**（「〜です」「〜ます」「〜でしょう」） |
| 1文の目安 | 60字以内。短く切る |
| 冒頭文（リード文） | **必須**。最初のH2の前に配置。4要素（悩み・結論・理由・ベネフィット）を含める |
| まとめ | **必須**。記事末尾に配置。結論の再提示＋次のアクションを示す |
| 結論 | 各H2の冒頭に必ず置く（結論ファースト） |
| 専門用語 | 使用可。初出時に1行で解説を添える |
| 感情表現 | 禁止（「かわいそう」「頑張って」など） |
| 装飾指定 | 本文には書かない。末尾の装飾メモに別記する |
| データ | 必ず出典を明記する |

### 出力フォーマット
- **形式**: Markdown
- **1行目**: `# タイトル`
- **タイトル直下**: HTMLコメントでメタ情報を記載
- **冒頭文（リード文）**: タイトル・メタ情報の直後、最初のH2の前に必ず配置
- **本文**: Markdown記法のみ（HTMLタグ・装飾タグは書かない）
- **装飾メモ**: 本文末尾に `---` で区切って別記
- **内部リンク候補**: 装飾メモの下に3〜5本（SEO記事タイトル形式）

### メタ情報フォーマット（タイトル直下に必ず記載）

```markdown
<!--
kw: メインキーワード
category: WordPressカテゴリ名
folder: 01_pillar / 02_articles / 03_data / 05_stock
status: draft
pillar_link: 紐づくピラー記事のslug
created: YYYY-MM-DD
tags: タグ1, タグ2, タグ3
-->
```

### 装飾メモフォーマット（本文末尾に必ず記載）

```markdown
---

## 装飾メモ（パートナー向け・SWELLで設置）

### 吹き出し
- 場所：[H2名]の直後
- アイコン：薬剤師
- 向き：右
- テキスト：[内容]

### ボックス
| 場所 | 色 | タイトル | 内容 |
|---|---|---|---|
| [H2・H3名] | 緑 / 青 / 赤 / 黄 | [タイトル] | [内容] |

### CTA（記事末尾）
- ボタン①：[テキスト]（URL：アフィリエイトURL）
- ボタン②：[テキスト]（URL：アフィリエイトURL）
- スタイル：横並び・クリック計測ON

### 内部リンク（記事公開後に設置）
- 場所：まとめの直後
- ① [記事タイトル]
- ② [記事タイトル]

---
```

## ⑧ 記事作成手順

```
【下書き前工程】

STEP 1: PMが構成・KW・タイトルを承認する

STEP 2: Claudeが競合上位記事を最終確認する
└ 上位3〜5記事の見出し・文字数・差別化ポイントを整理する

STEP 3: Claudeがデータ・根拠を収集する
└ 調剤報酬改定資料・決算IR・厚労省統計を確認する

STEP 4: 構成を最終調整する
└ 冒頭文の4要素（悩み・結論・理由・ベネフィット）を確認する
└ 各H2の結論を先出しできるか確認する

【執筆工程】

STEP 5: Claudeが本文を執筆する（付録Dのテンプレを使う）
└ Markdown形式で出力（ですます調）
└ 冒頭文・まとめを必ず含める
└ メタ情報・装飾メモ・内部リンク候補を末尾に付ける

【下書き後レビュー工程】

STEP 6: Claudeが自己チェック・推敲を行う

  ▼ SEOチェック
  - [ ] タイトルにメインKWが含まれているか
  - [ ] H2の最初1〜2個にメインKWまたは関連語が含まれているか
  - [ ] 本文全体でKWが自然に散りばめられているか（詰め込み過ぎない）
  - [ ] メタディスクリプション候補がKWを含む120文字以内か

  ▼ 文脈・論理チェック
  - [ ] 冒頭文で読者の悩みを正確に捉えているか
  - [ ] 各H2は結論ファーストになっているか
  - [ ] データ・根拠の出典が明記されているか
  - [ ] 転職を煽らず、読者が自己判断できる構成になっているか
  - [ ] まとめで結論を再提示し、次のアクションを示しているか

  ▼ 読みやすさチェック
  - [ ] 1文が60字以内に収まっているか（長い文を分割する）
  - [ ] 1段落が3〜4文程度になっているか
  - [ ] ですます調が統一されているか（断定調の混入がないか）
  - [ ] 冗長な表現を削除しているか（「〜ということになります」「〜かと思います」など）
  - [ ] 読んで引っかかる箇所がないか

  ▼ 推敲・リライト
  - [ ] 上記チェックで問題があった箇所を修正する
  - [ ] 修正後、全文を通読して違和感がないか最終確認する

STEP 7: PMが専門的事実確認・加筆修正を行う

STEP 8: CoworkがWP REST APIで下書き保存する（付録Eのテンプレを使う）

STEP 9: パートナーがSWELLで装飾メモを見ながら設置・装飾する

STEP 10: PMが最終確認して公開する
```

## ⑨ 内部リンク設計手順

### トピッククラスター構造

```
【まとめ記事・ピラーページ（01_pillar）】
↑ 集客記事・詳細記事からリンクが集まる
↓ 各詳細記事へリンクを張る（本文中・自然な流れで）

【集客記事・クラスターページ（02_articles / 05_stock）】
↑ データ記事・ニュース記事からリンクが来る
↓ まとめ記事・ピラーページへ必ずリンクを張る（相互リンク）

【データ記事（03_data）】
↓ 関連するまとめ記事・集客記事へリンク
```

**内部リンクのルール**:
1. 1記事に内部リンクは3〜7本（多すぎない）
2. アンカーテキストはKWを含む自然な文章にする（「こちら」は使わない）
3. まとめ記事と詳細記事は必ず相互リンクする
4. 新記事を公開するたびに、既存記事から新記事へのリンクも追加する

## ⑩ 収益導線設計手順

### 収益の流れ

```
検索流入記事（05_stock）← ロングテールKWで流入
└→ データ・解説記事（03_data）
└→ まとめ記事・ピラー（01_pillar）
└→ 転職判断記事（02_articles）
└→ キラーページ（02_articles）
└→ 転職サイトCTA → 登録
```

### CTAの設置ルール

| 記事タイプ | CTA設置場所 | CTA内容 |
|---|---|---|
| 05_stock（流入記事） | 記事末尾のみ | 「給与が妥当か気になる方へ」ソフトCTA |
| 03_data（データ記事） | 記事末尾＋関連リンク | 「実際の転職相談はこちら」 |
| 02_articles（判断記事） | 記事中盤＋末尾 | 「転職メリットがある場合はこちら」 |
| 01_pillar（まとめ記事） | 複数箇所 | 転職サイト比較・ランキング形式のCTA |

**CTAの鉄則（転職煽り禁止）**:
- 「今すぐ転職しよう」は書かない
- 「あなたの給与は低すぎる」は書かない

## ⑫ WordPress投稿フロー

### 使用ツール
**Cowork**（デスクトップアプリ）

### フロー

```
ClaudeがMarkdownで記事を出力（メタ情報・装飾メモ付き）
↓
GitHubの該当フォルダに保存（draft.md）
↓
CoworkにMarkdownと投稿設定を渡す（付録Eのテンプレを使う）
↓
CoworkがWP REST APIで下書き保存する
↓
パートナーがSWELLで装飾メモを見ながら設置・装飾する
↓
PMが管理画面で最終確認して公開する
```

### SWELLでパートナーが手動設置するブロック

| ブロック | 設置タイミング |
|---|---|
| ふきだしブロック | 下書き受け取り後 |
| SWELLボックス（色付き） | 下書き受け取り後 |
| CTAボタン（クリック計測付き） | 下書き受け取り後 |
| アイキャッチ画像 | 下書き受け取り後 |
| 関連記事ブロック（内部リンク） | 記事公開後 |

### セキュリティルール
- アプリケーションパスワードはCowork内でのみ使用する
- チャット（Claude.ai）には絶対に貼らない
- 3ヶ月ごとに再発行する

### アプリケーションパスワードの発行手順
1. WordPress管理画面にログイン
2. 左メニュー「ユーザー」→「プロフィール」
3. 下にスクロール →「アプリケーションパスワード」セクション
4. 名前欄に「Cowork」と入力 →「新しいアプリケーションパスワードを追加」
5. 表示されたパスワードをCoworkにのみ保存する（この画面を閉じると再表示不可）

## ⑬ リライト手順

リライトは記事を書くことと同様に重要な作業です。50記事以上になってきた段階で本格的に取り組みます。

### リライトの目的（優先順）
1. 検索順位を上げる
2. 回遊率を上げる
3. CTA転換率を上げる

### リライト手順

```
STEP 1: リライト候補を選ぶ
└ サーチコンソールで「表示されているがクリックされていない記事」を探す

STEP 2: 現状を確認する
└ どのKWで検索されているか
└ KWの検索意図と現在の記事内容がズレていないか

STEP 3: ライバルを再リサーチする

STEP 4: リライト内容を決めてClaudeに依頼する

STEP 5: 更新・公開後に効果測定する（最低1ヶ月）
```

**リライトの注意点**:
- 上位表示されている記事は触らない
- キーワードを無理に詰め込まない
- 内容が似た記事が増えたら統合を検討する
